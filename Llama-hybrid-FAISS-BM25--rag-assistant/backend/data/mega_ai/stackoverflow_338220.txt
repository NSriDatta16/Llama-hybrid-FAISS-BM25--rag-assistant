This looks like a particularly nasty deadlock. It's hard to tell for sure without a little more information. Here's what I see without the code in front of me, though: Possible deadlock on (or at least contention for) Object reference 0x00002aace2276720 between "Text Viewer Hover Presenter" and "Worker-3" while "Worker-3" is holding the lock on reference 0x00002aace2276ad0. This second reference seems to be the cause for a whole pile of blocked threads (specifically "Worker-4", "Worker-1" and "Worker-0"). My suggestion would be to check ASTProvider.java line 450 (see the stack trace for the first two where the suspicious object lock seems to be held but then doesn't seem to get past the wait()). I would also suggest try running the executable locked to a single core (assuming that this is a multicore system). The next piece of code that looks like it's worth testing would be SelectionListenerWithASTManager.java line 153 (where reference 0x00002aace2276ad0 is locked, causing the Workers to block).
