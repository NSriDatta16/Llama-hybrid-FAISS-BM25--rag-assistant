In addition to the multiple/no subscribers scenarios above, events are also used to reduce code coupling - eg method A() doesn't need to know anything about about method B() at compile time. This enables better separation of concerns and less fragile code. In the wild, you're more likely to see events used in framework and UI code, whereas within the domain logic of an application developers more often use things like Separated Interface and Dependency Injection to decouple code. Recently there has been a bit more discussion in various arenas regarding using events within domain logic, an approach which is cunningly named Domain Events .
