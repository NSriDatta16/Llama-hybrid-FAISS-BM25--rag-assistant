Typically speaking, "debugging" [ie attaching to a process and inspecting execution] is not viable - for many reasons not the least of which is data sensitivity [eg developers are rarely qualified\cleared to inspect the data we manipulate] So this usually comes down to inferring execution from secondary sources and artifacts. This then boils down to ... Logging, Logging, Logging, A large majority of software written these days falls into either of Java or .Net camps, so leverage log4j and log4net respectively. Also having a buller-proof Ops-centric configuration guide and validation process helps. Remember the people responsible for the hardware and environment rarely understand the configuration requirements of the applications they are hosting.
